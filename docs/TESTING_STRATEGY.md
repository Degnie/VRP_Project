# Estrategia de Pruebas (Testing Strategy)

> Reescrito para la arquitectura híbrida Python/C++ (post-migración, ver [CHANGELOG.md](../CHANGELOG.md) `0.1.0-alpha` en adelante). La versión anterior de este documento describía la GUI Qt/C++ académica; esa suite (`tests/test_core.cpp` + CTest) sigue existiendo en el árbol académico pero no es la suite activa de este SaaS.

## 1. Cobertura Exigida

* No se mide cobertura con una herramienta de coverage (no hay umbral numérico exigido por CI). La regla real es: **todo lo que toca una regla de negocio (validación de invariantes, cálculo de costos, persistencia) debe tener un test que falle si la regla se rompe**, no una métrica de porcentaje.
* Núcleo algorítmico C++ (`core_cpp/`: `Graph`, `CostMatrix`, `NearestNeighbor`, `SimulatedAnnealing`, operadores 2-opt/Or-opt/3-opt): cubierto indirectamente vía los tests Python de `service.solver_orchestrator`, que ejercitan el pipeline completo (construcción → optimización → validación) tanto con bindings C++ activos como con el fallback puro Python.
* Modelos de dominio Python (`backend_python/models.py`: `Coordinate`, `Cliente`, `Deposito`, `Flota`, `Instancia`, `Ruta`, `Solucion`): 100% de los invariantes declarados (demanda > 0, demanda entera, capacidad no excedida, IDs de cliente únicos, cada cliente visitado exactamente una vez) están cubiertos por `tests/unit/test_models.py` — incluye `test_cliente_demanda_debe_ser_entera` (`0.3.2`), regresión del invariante introducido en `0.3.1`.
* Estado actual: 59 tests pasando, 5 skipped por diseño (2 dependen de que ambas bases de datos estén disponibles simultáneamente, 1 depende de bindings C++ compilados, 2 dependen de un servicio OSRM real; ver sección 2).

## 2. Estrategias por Capa

* **Modelos de dominio (`backend_python/models.py`):** tests unitarios puros, sin I/O. Validan invariantes vía construcción (`dataclass` con `__post_init__` que lanza `ValueError`/`AssertionError`) — el test intenta construir un objeto inválido y espera la excepción.
* **Orquestador (`backend_python/service/solver_orchestrator.py`):** tests de integración que resuelven instancias de ejemplo y verifican que la `Solucion` resultante sea válida (todos los clientes visitados, capacidad respetada, costo > 0). Corre igual con o sin bindings C++ compilados — si `vrp_solver` no está disponible, el orquestador cae automáticamente al fallback Python y los tests deben pasar en ambos modos. Desde `0.4.0`, la matriz de costos (`_build_cost_lookup()`) se construye una sola vez antes de esa bifurcación, garantizando que ambos caminos usen la misma fuente de distancias (OSRM o euclídea) — este es el mismo estándar de paridad, extendido a la fuente de datos, no solo al algoritmo.
* **Cliente OSRM (`backend_python/service/osrm_client.py`):** tests de integración contra un servicio OSRM real (`osrm/osrm-backend` vía Docker, mapa pre-procesado con `make osrm-prepare`). Se activan solo si `OSRM_URL` está definida en el entorno; si no, se saltan (`skipif`) — mismo patrón exacto que Postgres/Mongo. El caso de host inalcanzable (`OSRMError` propagada) no depende de un servicio real y corre siempre.
* **Persistencia (`backend_python/persistence/postgres_adapter.py`, `mongodb_adapter.py`):** tests de integración contra contenedores Docker reales (`postgres:15`, `mongo:7`), no contra mocks. Se activan solo si `DATABASE_URL`/`MONGO_URL` están definidas en el entorno (vía `.env.local`, no versionado); si no están, los tests correspondientes se saltan (`skipif`) en vez de fallar — así la suite corre en cualquier máquina sin Docker, pero exige DB real para validar la capa de persistencia de verdad.
* **API REST (`backend_python/api/__init__.py`):** tests de integración vía `starlette.testclient.TestClient` sobre la app creada por `create_app()`, ejercitando los endpoints reales (`/solve`, `/instances`, `/solutions/{id}`, `/health`) contra las mismas DBs Docker.
* **Núcleo C++ puro (`core_cpp/`):** sin suite de tests C++ dedicada en la arquitectura actual (a diferencia del árbol académico, que sí tenía `test_core.cpp` + CTest). La validación del núcleo ocurre a través de los tests Python del orquestador, que ejercitan los bindings pybind11 de punta a punta. Se considera deuda técnica documentada, no un olvido — ver sección 4.

## 3. Inyección de Fallos (Fault Injection)

* **Directiva crítica: matrices de costo malformadas.** Como el núcleo asume grafos dirigidos con matrices de adyacencia asimétricas (ver [docs/adr/0002-asymmetric-cost-matrices.md](adr/0002-asymmetric-cost-matrices.md)), cualquier cambio a `CostMatrix` o a los builders/optimizers debe probarse con matrices intencionalmente malformadas: dimensiones inconsistentes, valores negativos, y sobre todo **matrices asimétricas reales** (`costo[i][j] != costo[j][i]`) para confirmar que ningún algoritmo asume `costo[i][j] == costo[j][i]` por accidente — ese es el bug de mayor impacto posible en este dominio, porque pasaría silenciosamente en instancias euclidianas simétricas y solo se manifestaría con datos reales (tráfico, calles de un solo sentido).
* **Instancias inviables (demanda total > capacidad de flota):** `Instancia.__post_init__` valida esto en construcción y lanza excepción antes de llegar al solver — cubierto en `test_models.py`.
* **Fallo de conexión a base de datos:** los adapters (`PostgreSQLAdapter`, `MongoDBAdapter`) reintentan la conexión inicial hasta 3 veces con 1 segundo de espera fija entre intentos (`0.3.6`) antes de lanzar `ConnectionError`, en vez de fallar en el primer intento. `PostgreSQLAdapter` usa `connect_timeout=5` explícito (añadido en `0.3.6` tras detectar que sin él un intento contra un host inalcanzable podía colgarse indefinidamente en Windows); `MongoDBAdapter` ya tenía `serverSelectionTimeoutMS=5000`. Verificado manualmente contra un puerto cerrado real para ambos adapters — ver CHANGELOG `0.3.6`.
* **Archivos/payloads corruptos en la API:** validación vía modelos Pydantic en los endpoints; un payload que no matchea el schema devuelve `422` antes de tocar el orquestador o las DBs.
* No hay fuzzing automatizado de matrices ni de payloads HTTP — es una oportunidad futura, no algo implementado hoy.
* **OSRM no disponible/timeout (`0.4.0`):** `SolverOrchestrator._build_cost_lookup()` cae silenciosamente a distancia euclídea si `OSRM_URL` no está configurado (salta la llamada HTTP, sin latencia) o si la llamada falla (timeout, error HTTP, código de error de OSRM) — nunca bloquea `/solve`. Cubierto en `tests/unit/test_optimizers.py::TestCostMatrixFallback` con ambos casos (no configurado, configurado pero inalcanzable vía `monkeypatch`).
* **Número de rutas excede vehículos disponibles:** `Instancia.__post_init__` valida demanda total contra capacidad agregada (`num_vehiculos * capacidad`), pero eso no garantiza que un bin-packing greedy (NN) reparta esa demanda en `num_vehiculos` rutas o menos. `SolverOrchestrator.solve()` (`0.3.3`) valida `len(solution.rutas) > flota.num_vehiculos` tras obtener la solución de cualquiera de los dos caminos (C++ o fallback Python) y lanza `ValueError` — cubierto en `tests/unit/test_optimizers.py::TestFleetSizeValidation` con un caso construido explícitamente (demanda cabe en agregado, pero greedy fragmenta en más rutas que vehículos).
* **Caso pendiente concreto: colisión de ID cliente/depósito.** `Graph::add_node` (core_cpp/include/graph.hpp) no detecta colisión de IDs, solo valida rango — un cliente con `id=0` sobrescribe silenciosamente el nodo del depósito en el grafo C++ (que siempre lo reserva en `id=0`). El fix aplicado en `0.3.1` (IDs de cliente desde `1` en `api/__init__.py`) corrige el único punto de entrada actual, pero `Graph::add_node` en sí sigue sin protección propia. Es el primer candidato a cubrir si se implementa la suite C++ dedicada (ver sección 4): un test que llame `add_node(0, ...)` dos veces y espere una excepción, en vez de una sobrescritura silenciosa.

## 4. Decisiones Históricas y Deuda Técnica

* **Cola de mensajería para `/solve` asíncrono:** descartada. Ver CHANGELOG, sección "Rechazado / Descartado".
* **ORM para el adapter de PostgreSQL:** descartado. Ver CHANGELOG, sección "Rechazado / Descartado".
* **Retry/backoff en adapters de persistencia:** implementado en `0.3.6` (backoff fijo, 3 intentos, solo en conexión inicial) tras reconsiderar la decisión original de `0.3.0` tras el rechazo — el despliegue futuro en una máquina distinta a la actual hace que un fallo transitorio de red deje de ser hipotético. Ver CHANGELOG `0.3.6`.
* **Suite de tests C++ dedicada (`core_cpp/`):** actualmente el núcleo C++ solo se valida indirectamente vía los tests Python del orquestador. Es deuda técnica real y documentada — si el núcleo crece (más operadores, paralelización), esta indirección dejará de ser suficiente y se necesitará una suite C++ nativa (Catch2/GoogleTest) equivalente a la que existía en el árbol académico.
* **Bindings C++ no compilados en el entorno de desarrollo actual:** el proyecto corre en fallback Python puro. No es deuda de diseño (el fallback es intencional y está probado), pero sí significa que el camino de bindings pybind11 no se ejercita en CI hasta que haya un entorno con Python 64-bit + headers de desarrollo disponibles.
* **`psycopg2` vs `psycopg` (v3):** el adapter de PostgreSQL intenta `psycopg2` primero y cae a `psycopg` (v3) si no está disponible, porque `psycopg2` no publica wheels para Python 3.14+. Es una decisión pragmática de compatibilidad, no una preferencia de diseño — si el entorno de producción fija una versión de Python con soporte estable de `psycopg2`, ambas rutas siguen funcionando sin cambios.
* **`Graph::add_node` sin detección de colisión de IDs (`core_cpp/include/graph.hpp`):** solo valida rango (`0 <= id < n_nodes`), no si la posición ya fue asignada — una segunda llamada con el mismo `id` sobrescribe el nodo anterior sin error. Detectado en auditoría (`0.3.1`) por la colisión cliente/depósito (ver sección 3). El fix aplicado corrige el único punto de entrada actual (`api/__init__.py` genera IDs desde `1`), pero el método en sí sigue sin protección propia — deuda técnica conocida y no bloqueante, ya que hoy no hay otro caller que pueda disparar la colisión.
* **`vrp_solver.Graph()` construido con tamaño incorrecto — corregido en `0.3.2` (bug de Fase 2, no introducido por `0.3.1`).** `solver_orchestrator.py:127` pasaba `self.instance.flota.num_vehiculos` como tamaño del grafo en vez de `1 + len(self.instance.clientes)` (depósito + clientes). Con más clientes que vehículos (caso normal), `add_node` habría lanzado `std::out_of_range` en cuanto se compilaran los bindings C++. Corregido, pero **sigue sin existir un test que ejercite `_solve_cpp_pipeline` end-to-end** — el fix reduce el riesgo de que el pipeline C++ falle al activarse, y desde `0.3.3` el caso de "más rutas que vehículos" sí tiene cobertura parcial vía el fallback Python (`TestFleetSizeValidation`, mismo código de validación en `SolverOrchestrator.solve()` que se ejecutaría también para el resultado del pipeline C++). El camino C++ específico (`_solve_cpp_pipeline`, incluyendo `NearestNeighbor::solve()` en sí) sigue sin cobertura directa hasta que existan bindings compilados en este entorno.
* **Límite de tamaño de matriz de OSRM y solución de chunking (`0.4.0`, corregido en `0.4.1`).** El endpoint `/table` de OSRM tiene un límite práctico de coordenadas por request (configurable en el servidor, no ilimitado) — en conflicto directo con la promesa de escala del README ("50 a 100k+ clientes"). Se implementó chunking desde el inicio (`osrm_client.py::get_osrm_matrix`, partición en bloques). Bug encontrado en auditoría (`0.4.1`): el tamaño de cada bloque combinaba fila+columna sin dividir `max_table_size` entre ambos, permitiendo requests hasta `2x` el límite configurado — corregido usando `max_table_size // 2` como tamaño de bloque. Deuda técnica restante: el chunking no se ha probado contra instancias de miles de clientes reales (solo contra el caso sintético de 4 coordenadas en `test_osrm_client.py`) — el rendimiento del ensamblaje de una matriz muy grande (miles × miles) en memoria Python no está validado a esa escala.
* **Cobertura geográfica limitada a Perú/Lima Metropolitana (`0.4.0`).** El mapa OSRM se prepara a partir del extracto de Perú de Geofabrik (no hay uno más granular para Lima). Instancias con coordenadas fuera de ese extracto obtendrían resultados incorrectos o un error de OSRM (que activaría el fallback a euclídea, no un fallo silencioso de datos incorrectos — comportamiento seguro, pero limitado en cobertura). Ampliar a otras regiones es trabajo futuro, no bloqueante para el alcance actual.
